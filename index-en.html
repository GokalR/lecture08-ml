Properties

Problem: read without control

A regular public attribute in Python returns the value "as is". On a read, the value is simply returned from memory. There is nowhere to insert a check, an audit log entry, or a recomputation.

class CreditCard: def __init__(self, pan: str): self.pan = pan # обычный атрибут card = CreditCard("8600112233445566") print(card.pan) # весь номер виден целиком
8600112233445566

We would like reading card.pan to show only part of the number. But a regular attribute cannot do that: it does not execute code.

Properties

Solution: the @property decorator

@property turns a method into a "virtual field". From the outside an object is read as an attribute, without parentheses. But Python actually calls the method, and inside the method you can run any code.

class CreditCard: def __init__(self, pan: str): self._pan = pan # _pan - внутреннее хранилище @property def pan(self) -> str: # это метод... return self._pan card = CreditCard("8600112233445566") print(card.pan) # ...но читаем без скобок, как поле

Analogy: instead of a key to the vault (direct field access) we put a service window (a method). From the outside it is just as convenient, inside everything stays under control.

Properties

Why turn an attribute into a property

A getter that simply returns a value is useless on its own. A property is useful when work must happen when the field is accessed. Three typical reasons:

1

Logic on read

Each access runs code: masking, audit logging, validation.

2

Computed values

The value is not stored; it is computed on the fly from the object's current data.

3

Interface compatibility

A public attribute can become a property without changes to external code.

Properties · Reason 1

Logic on read

We add code inside the property. On every read it masks the card number and records the access in the security log.

class CreditCard: def __init__(self, client_name: str, raw_pan: str): self.client_name = client_name self._pan = raw_pan @property def masked_pan(self) -> str: print("[Аудит] Чтение номера карты") return f"{self._pan[:4]} **** **** {self._pan[-4:]}" card = CreditCard("Arman", "8600112233445566") print(f"Номер карты: {card.masked_pan}")
[Аудит] Чтение номера карты Номер карты: 8600 **** **** 5566

Properties · Reason 2

Computed properties

A property does not store a value; it recomputes it on every access from the object's current data. If the data changes, the result changes too.

class LoanPortfolio: def __init__(self, principal: float, interest_rate: float): self.principal = principal self.interest_rate = interest_rate @property def total_debt(self) -> float: """Пересчитывается при каждом обращении""" return self.principal * (1 + self.interest_rate) portfolio = LoanPortfolio(10_000_000, 0.20) print(f"Итого к возврату: {portfolio.total_debt} сум") portfolio.principal -= 2_000_000 print(f"После погашения части: {portfolio.total_debt} сум")
Итого к возврату: 12000000.0 сум После погашения части: 9600000.0 сум

Properties · Reason 3

Interface compatibility: Python vs Java

Java

Boilerplate from day one

Every field is wrapped right away in getBalance() and setBalance(), even when there is no logic inside yet.

Python

Only when needed

Start with a regular attribute. When logic is needed, turn it into a @property. External code keeps working without changes.

That is why getters are not written in advance in Python. A regular attribute only becomes a property when logic appears.

Properties · Question breakdown

"Why a getter if the field is still reachable?"

Python has no hard privacy: the _pan field can be reached directly and even changed. The underscore _ does not technically forbid anything.

card = CreditCard("Arman", "8600112233445566") print(card._pan) # 8600112233445566 - читать Python не мешает card._pan = "0000000000000000" # изменить тоже даёт

The underscore _ works as a convention, not a lock. It says: "the field is internal, do not touch it from outside". But physically the access is open.

Properties · Question breakdown

The getter sets the main path

The point of a getter is not to make access impossible. The point is that the right path is the most convenient one. Then ordinary code goes through it naturally.

Getter card.masked_pan

Main entrance

All normal code goes through it. Masking, audit logging and checks turn on automatically.

Field card._pan

Service door

Technically open. But reaching for it from outside means deliberately breaking the agreement, and that shows up in code review.

Properties · Question breakdown

A getter hides the internal layout

Imagine: in six months the rules change. The card number can no longer be stored in the open, it must be encrypted. We change the class's internal layout:

# Было: номер хранится как есть self._pan = raw_pan # Стало: номер хранится зашифрованным self._encrypted_pan = encrypt(raw_pan) # Геттер меняем в одном месте, имя свойства прежнее: @property def masked_pan(self) -> str: real = decrypt(self._encrypted_pan) return f"{real[:4]} **** **** {real[-4:]}"

Code that read card.masked_pan keeps working without a single edit. Code that took card._pan directly would break everywhere: the field no longer exists.

Properties

Controlling writes: the setter

So far @property controls reading only. To intercept writes, that is, assignments via =, you add a setter to the property: a method decorated with @name.setter.

class Account: def __init__(self, balance: float): self._balance = balance @property def balance(self): # чтение: acc.balance return self._balance @balance.setter def balance(self, value): # запись: acc.balance = ... self._balance = value

Getter and setter share the same name balance. Now the field can be both read and assigned, but the write goes through a method.

Properties

A setter with data validation

The main benefit of a setter: validate the value before saving it. If it is not acceptable, the setter raises an error and the field stays unchanged.

class CreditLimitManager: def __init__(self, current_limit: float): self._limit = current_limit @property def limit(self) -> float: return self._limit @limit.setter def limit(self, new_value: float): if new_value < 0: raise ValueError("Лимит не может быть отрицательным") self._limit = new_value manager = CreditLimitManager(50_000_000) try: manager.limit = -100 except ValueError as error: print(f"Запись отклонена: {error}")
Запись отклонена: Лимит не может быть отрицательным

Under the hood

Getter and setter: one property

The getter and the setter together form a single property. Python picks the right method itself: are we reading the field, or assigning to it.

@property создаёт свойство 'limit' с логикой чтения @limit.setter к тому же свойству добавляет логику записи manager.limit Python вызывает геттер manager.limit = 700000 Python вызывает сеттер со значением 700000

The names of the getter and the setter must match. That is why the decorator is written as @limit.setter: it attaches to the already-created limit property.

Properties

Example: a secure PIN code

The getter does not hand the real PIN to the outside, and the setter validates the format on every write attempt.

class PinValidator: def __init__(self): self._pin = None @property def pin(self) -> str: return "****" # реальное значение не отдаём @pin.setter def pin(self, value: str): if not value.isdigit() or len(value) != 4: raise ValueError("ПИН-код: ровно 4 цифры") self._pin = value print("ПИН-код обновлён") card = PinValidator() card.pin = "1234" print("ПИН на экране:", card.pin)
ПИН-код обновлён ПИН на экране: ****

Properties

A property without a setter: read-only

If a property has no setter, it cannot be assigned to: Python raises an error. This is how you protect fields that must not change after creation.

class Contract: def __init__(self, contract_id: str, client: str): self._contract_id = contract_id self.client = client @property def contract_id(self) -> str: return self._contract_id # сеттера нет contract = Contract("UZ-2024-001", "Тимур") print(contract.contract_id) try: contract.contract_id = "UZ-2024-999" except AttributeError: print("Защита сработала: номер контракта менять нельзя")
UZ-2024-001 Защита сработала: номер контракта менять нельзя

Properties

Reference BankAccount class

Putting it all together: protected balance, read through a property, and a deposit method with validation.

class BankAccount: def __init__(self, owner: str, initial_balance: float = 0.0): self.owner = owner self._balance = initial_balance # защищённый баланс @property def balance(self) -> float: return self._balance def deposit(self, amount: float) -> None: if amount <= 0: print("Ошибка: сумма пополнения должна быть больше нуля") return self._balance += amount

The balance is closed off from direct writing: read through @property, changed only through methods.

Properties

BankAccount: withdrawing funds

The same class. A withdraw method with two checks: amount and remaining balance.

def withdraw(self, amount: float) -> None: if amount <= 0: print("Ошибка: сумма списания должна быть положительной") return if amount > self._balance: print("Транзакция заблокирована: недостаточно средств") return self._balance -= amount acc = BankAccount("Дмитрий", 1_000_000.0) acc.withdraw(1_500_000.0)
Транзакция заблокирована: недостаточно средств

The method itself checks the amount and the remaining balance. An invalid operation simply does not go through.

Class level

Instance attribute vs class attribute

An instance attribute is created via self.field separately for each object. A class attribute is declared in the class body - one copy for all instances.

class NationalBankCreditProduct: base_interest_rate = 0.20 # атрибут класса - общий для всех def __init__(self, client_name: str, loan_amount: float): self.client_name = client_name # атрибут экземпляра self.loan_amount = loan_amount contract1 = NationalBankCreditProduct("Тимур", 50_000_000) contract2 = NationalBankCreditProduct("Елена", 90_000_000) print(f"Ставка для Тимура: {contract1.base_interest_rate * 100}%") print(f"Ставка для Елены: {contract2.base_interest_rate * 100}%")
Ставка для Тимура: 20.0% Ставка для Елены: 20.0%

Class level

Changing a class attribute affects everyone

print("--- До изменения ставки ---") print(f"Тимур: {contract1.base_interest_rate * 100}%") print(f"Елена: {contract2.base_interest_rate * 100}%") NationalBankCreditProduct.base_interest_rate = 0.22 print("--- После изменения ставки ---") print(f"Тимур: {contract1.base_interest_rate * 100}%") print(f"Елена: {contract2.base_interest_rate * 100}%")
--- До изменения ставки --- Тимур: 20.0% Елена: 20.0% --- После изменения ставки --- Тимур: 22.0% Елена: 22.0%

If you change it through an instance (contract1.base_interest_rate = 0.25), Python creates a new attribute on that instance - the class attribute stays the same.

Class level

Why we need @classmethod

A class has one constructor, the __init__ method. It expects ready-made fields sorted into place. But data often arrives in a different shape: JSON from an API, a row from a CSV file, the current time.

import json # Ответ API пришёл одной JSON-строкой: raw = '{"amount": 150000, "currency": "UZS", "ts": "2024-01-15"}' # Перед созданием объекта строку приходится разбирать вручную: data = json.loads(raw) t = Transaction(data["amount"], data["currency"], data["ts"])

If the object is created this way in ten places across the project, JSON parsing is repeated ten times. Creation logic gets "smeared" across the code.

Class level

Solution: @classmethod

A method decorated with @classmethod acts as a second, named constructor. As its first argument it receives not the self object, but the class itself, cls.

Regular method

Receives self

The method gets a specific object. It works with that object's data, for example self.amount.

Class method

Receives cls

The method gets the class itself. A cls(...) call creates a new object, just like Transaction(...).

The parsing logic is hidden inside the class method. A short call remains on the outside: Transaction.from_json(raw).

Class level

Factory method: from JSON

Let's put the solution together in code. The from_json method parses the string inside itself and returns a ready object via cls(...).

import json class Transaction: def __init__(self, amount: float, currency: str, timestamp: str): self.amount = amount self.currency = currency self.timestamp = timestamp @classmethod def from_json(cls, json_str: str) -> 'Transaction': data = json.loads(json_str) # разбираем ответ API return cls(data["amount"], data["currency"], data["ts"]) def __str__(self): return f"{self.amount} {self.currency} @ {self.timestamp}" t1 = Transaction.from_json('{"amount": 150000, "currency": "UZS", "ts": "2024-01-15"}') print(t1)
150000 UZS @ 2024-01-15

@classmethod receives cls and creates the object via cls(...) - this is a second way to build a Transaction.

Class level

Constructors from CSV and from time

Two more factory methods on the same Transaction class:

@classmethod def from_csv_row(cls, csv_line: str) -> 'Transaction': parts = csv_line.strip().split(",") # из строки CSV return cls(float(parts[0]), parts[1], parts[2]) @classmethod def create_now(cls, amount: float) -> 'Transaction': now = datetime.now().isoformat() # текущее время return cls(amount, "UZS", now) t2 = Transaction.from_csv_row("320000.0,UZS,2024-01-15T11:00:00") t3 = Transaction.create_now(75000.0) print(t2) print(t3)
320000.0 UZS @ 2024-01-15T11:00:00 75000.0 UZS @ 2024-03-20T14:22:31

One class - several entry points: from JSON, from CSV, from the current time.

Class level

cls vs hard-coded class name in inheritance

Inside a factory method, the object is created via cls(...), not by the class name directly. Then the method also works correctly in subclasses.

class BaseCard: def __init__(self, number: str): self.number = number @classmethod def from_masked(cls, masked: str) -> 'BaseCard': fake = masked.replace("****", "0000") return cls(fake) # cls, а не BaseCard - иначе потомки сломаются def __str__(self): return f"{self.__class__.__name__}: {self.number}" class GoldCard(BaseCard): pass print(BaseCard.from_masked("8600 **** **** 5566")) print(GoldCard.from_masked("8600 **** **** 5566"))
BaseCard: 8600 0000 0000 5566 GoldCard: 8600 0000 0000 5566

With BaseCard(...) instead of cls(...) the gold object would end up as a BaseCard instance, not a GoldCard.

Class level

Why we need @staticmethod

Sometimes there is a function that logically belongs to a class but does not work with a specific object or with the class itself. It needs neither self nor cls.

Example: validating phone number format next to the PhoneValidator class. It is a pure utility: takes a string, returns a string. You could move it to a plain function, but then it loses the connection to the class it belongs to.

The @staticmethod decorator lets you place such a utility inside the class. It is grouped near related code, but receives neither the object nor the class.

Class level

Static methods: @staticmethod

A method with @staticmethod is declared without self and cls. It is called through the class. Here it is a utility that brings a phone number into a single format.

class PhoneValidator: @staticmethod def clean_phone(phone_str: str) -> str: """Приводит номер к стандартному формату""" cleaned = "".join(c for c in phone_str if c.isdigit()) if cleaned.startswith("8"): cleaned = "7" + cleaned[1:] return f"+{cleaned}" formatted = PhoneValidator.clean_phone("8 (90) 123-45-67") print(f"Результат: {formatted}")
Результат: +79012345667

Class level

Why not just a global function?

@staticmethod groups utilities inside the class, exports them together with it, and allows overriding in subclasses.

class UzbekBankValidator: @staticmethod def is_valid_pinfl(pinfl: str) -> bool: """Проверяет формат ПИНФЛ (14 цифр)""" return pinfl.isdigit() and len(pinfl) == 14 @staticmethod def mask_card_number(pan: str) -> str: """Маскирует номер карты для безопасного вывода""" digits = pan.replace(" ", "") return f"{digits[:4]} **** **** {digits[-4:]}" print(UzbekBankValidator.is_valid_pinfl("12345678901234")) print(UzbekBankValidator.mask_card_number("8600 1122 3344 5566"))
True 8600 **** **** 5566

Class level

Three method types: regular method

A regular method takes self and works with the data of a specific object:

class BankAccount: transaction_fee = 0.01 # атрибут класса def __init__(self, owner: str, balance: float): self.owner = owner self._balance = balance def withdraw(self, amount: float) -> None: # обычный метод total = amount + amount * BankAccount.transaction_fee if total > self._balance: print("Недостаточно средств") return self._balance -= total print(f"Списано {amount} + комиссия. Остаток: {self._balance}") acc = BankAccount("Алия", 500_000.0) acc.withdraw(200_000.0)
Списано 200000.0 + комиссия. Остаток: 298000.0

Class level

Three method types: class and static

The same class - a class method (receives cls) and a static method (no context):

@classmethod def open_with_bonus(cls, owner: str) -> 'BankAccount': # cls создаёт объект - альтернативный конструктор return cls(owner, 100_000.0) @staticmethod def validate_transfer_amount(amount: float) -> bool: # утилита: ни self, ни cls не нужны return 0 < amount <= 1_000_000_000 bonus_acc = BankAccount.open_with_bonus("Серик") print(f"Баланс Серика: {bonus_acc._balance} сум") print(BankAccount.validate_transfer_amount(-100))
Баланс Серика: 100000.0 сум False

Protocols

Protocols and magic methods

Dunder methods (from Double Underscore) let user-defined objects integrate with Python's standard operators and functions.

By overriding __add__, __eq__, __len__ and others, you define how objects are added, compared and how they behave inside the language's standard constructs.
